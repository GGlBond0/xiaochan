# 监控抢单多账号多平台优先级轮询

## Goal

监控命中自动抢单从"单账号单次抢"升级为"多账号 + 多平台 + 优先级 + 失败换号/降级"。
解决痛点：门店限频按 (账号×门店) 计（上游 code==70，文案"您在该店铺下单超过店铺限制，今日已无法参加该活动"），单账号被限后当天无法再抢该门店任何平台任何活动；需用多个账号轮换抢同一个名额，并按用户设定优先级决定先抢哪个平台/活动。

## 背景（已实测确认）

- 限频语义：限频键 = (账号 × 门店)，**跨活动、跨平台共享**。账号1在门店A抢过一次 → 账号1被锁（按天/N天，文案"今日"），该门店所有平台所有活动都返回 code==70；不影响账号2/3 各自独立计算。
  - 实锤 HAR：`har/ProxyPin7-17_04_34_29.har`，同一 silk_id(222559356) 对 promotion_id=118672828(store_platform=3 京东) 与 118672829(store_platform=2 饿了么) 均返回 code==70。
- 限频 code==70 全平台一致（美团 GrabPromotionQuota / 饿了么·京东 OrderExchange 共用 status.code 结构）。用户已确认美团同为 code==70。
- 账号（login_state）**无平台属性**：一个小蚕账号的美团/饿了么/京东活动都能用同一登录态抢。账号多选与平台无关，不按平台过滤账号。
- 现有 `grabPlatforms` 是平台白名单（`AutoGrabServiceImpl:72` 按 store.type 过滤），决定"抢不抢"。
- 现有重试逻辑（`GrabServiceImpl:316`）只对 code==4(未开始)重试，其它失败直接 break 结束 —— 无法换号，是本次要改的核心。
- 现有定时抢分支（`AutoGrabServiceImpl` 未到点→建 auto=0 定时任务到 start 时间、注册 cron）与立即抢分支（时段内→异步 doGrab）—— 本次的"等待高优先级"复用定时抢机制。

## Requirements

### 配置侧（monitor_config）

- R1 抢单账号：`grabLoginStateId`（单 Integer）→ 扩为**可排序多选**（逗号分隔 id 串，顺序即账号优先级）。存量单值向后兼容（旧单值视为单元素列表）。
- R2 抢单平台：复用 `grabPlatforms`，**顺序即平台优先级**（逗号串先后 = 优先级高到低）。存量兼容（旧无序集合视为优先级即勾选顺序，默认仅美团）。
- R3 抢单模式：新增字段（枚举 SINGLE / ALL）。
  - SINGLE：抢一个名额。
  - ALL：每个勾选账号各抢一个名额。
- R4 `autoGrab=false` 时抢单相关字段（账号多选/平台/模式）一律忽略、置空（沿用现有 autoGrab 门禁）。

### 运行侧（命中后处理）

- R5 平台过滤+排序：命中产生的多个 (活动 promotionId, 平台 store.type) 组合，只保留勾选平台对应的组合，并按平台优先级排序。
- R6 SINGLE 模式：
  - 降级范围 = **同一次命中里同门店的所有 (活动,平台) 组合**（跨活动、跨平台，按 storeId 分组）。这些组合共享该门店名额，按平台优先级排序后逐级降级。不同门店之间不互相降级（不同门店是不同名额）。
  - 取最高优先级组合。未到该组合 start → 建定时任务到其 start（复用现有定时抢机制），**不抢、不碰低优先级组合**。
  - 已到 start → 立即抢；组合内按账号优先级依次尝试，**成功即停**。
  - 组合失败（所有账号都试过仍未成功）→ **降级到下一优先级组合**（未到点建定时、到点立即抢），直到成功或全部组合耗尽。
  - 优先级为硬约束：等待最高优先级期间，即使低优先级组合已开始也不抢。
- R7 ALL 模式：每个勾选账号**并行**独立、**不换号**——每个账号各自在所有命中门店按平台优先级抢，某账号在某门店失败（如 code==70 被该门店限频）即放弃该门店、继续下一门店，不换别的账号替抢。账号间互不影响（限频按账号计，不互相挡），各拿各的。组合间降级仍适用（同门店不同组合是不同名额）。目标：每账号尽量在每个门店各得1个名额。
- R8 换号 / 降级判定表（按上游 status.code）：

  | code | 含义 | 动作 |
  |---|---|---|
  | 0 | 成功 | 停（本账号/本组合完成） |
  | 70 | (账号×门店)限频 | 换下一账号 |
  | 6 | 活动名额已抢完 | 该组合失败 → 降级下一组合（换号无用） |
  | 4 | 未开始 | 等待/定时到 start（不应硬出现） |
  | -1 饭票不足（本地预判） | 该账号无饭票 | 换下一账号 |
  | -1 登录态过期/缺失 | 账号不可用 | 换下一账号 |
  | -1 活动详情缺失/商家黑名单/位置无效 | 组合本身不可抢 | 该组合失败 → 降级下一组合 |
  | 其它未知 | — | 该组合失败 → 降级下一组合（保守，避免高频打未知失败被风控） |

- R9 不做本地 (账号×门店) 3天/按天预判去重；接受 code==70 的无效请求（用户明确接受）。

### 兼容性

- R10 存量 monitor_config：单 `grabLoginStateId` → 当作单元素账号列表；`grabPlatforms` 无序 → 当作"全部勾选等优先级"（实际按串内顺序）；无模式字段 → 默认 SINGLE（与现有单账号单次行为一致）。
- R11 手动抢 / 定时抢（非监控触发）链路本次不改 doGrab 的对外契约，优先级/多账号/降级仅作用于监控自动抢（AutoGrabServiceImpl）链路。doGrab 仍按单个 config（单账号）执行；多账号轮换在 AutoGrab 层组装多次调用。

## Acceptance Criteria

### 配置侧
- [ ] AC1 监控配置页：抢单账号控件为可排序多选（拖拽/上下移排序），保存为有序 id 串，加载反显顺序正确。
- [ ] AC2 抢单平台控件为可排序多选，顺序即优先级，保存到 `grabPlatforms`（有序），加载反显正确。
- [ ] AC3 抢单模式 SINGLE/ALL 可选，保存到新字段，反显正确。
- [ ] AC4 `autoGrab=false` 时抢单字段不校验不落库（置空），保存成功。
- [ ] AC5 存量配置（单账号、无序平台、无模式）加载不报错，行为与上线前一致（单账号单次抢）。

### 运行侧 SINGLE
- [ ] AC6 命中多组合时，按平台优先级仅对最高优先级组合处理；未到 start → 建定时任务到 start，不抢低优先级。
- [ ] AC7 最高优先级组合已开始 → 立即抢；账号按优先级依次试，成功即停。
- [ ] AC8 账号返回 code==70 → 换下一账号；所有账号都 70/失败 → 降级下一优先级组合。
- [ ] AC9 组合返回 code==6（抢完）→ 直接降级下一组合，不在账号间换号。
- [ ] AC10 等待最高优先级期间，低优先级组合已开始也不抢（硬约束，日志可验证）。
- [ ] AC11 全部组合耗尽仍未成功 → 推送失败通知（含尝试概要），不无限重试。

### 运行侧 ALL
- [ ] AC12 ALL 模式每个勾选账号**并行**独立、不换号：各账号各自在所有命中门店按平台优先级抢，账号A 成功不影响账号B 继续。
- [ ] AC13 ALL 模式某账号在某门店遇 code==70（被该门店限频）→ 该账号放弃该门店、继续下一门店，**不换别的账号替抢**。
- [ ] AC13a ALL 模式下若命中多门店，每个账号在每个门店各按优先级尝试，目标每账号每门店最多1个名额（验证日志可见账号×门店维度的独立尝试）。

### 兼容/回归
- [ ] AC14 autoGrab=false 配置命中只通知、不产生任何抢单任务（回归零影响）。
- [ ] AC15 手动抢/定时抢（非监控）行为不变。

## Notes

- 限频 code==70 实锤见 `har/ProxyPin7-17_04_34_29.har`（entries 30/121）。
- ALL 模式下"每个账号独立跑 SINGLE" 中"账号优先级"的角色需在 design 明确：ALL 是每账号各抢一个名额，账号优先级在 ALL 下是否还有意义、如何取候选账号，需设计阶段定（候选 = 该账号自身？还是仍按账号优先级列表轮换？倾向：ALL 下每个账号就是它自己，账号优先级列表在 ALL 下退化为"参与账号清单"，顺序无额外意义）。
- 本地预判 (账号×门店) 去重不做（R9），但 design 仍需考虑 code==70 命中后是否本地标记该(账号,门店)短期不可用，避免同次命中内重复打 70（可选优化，非必需）。
