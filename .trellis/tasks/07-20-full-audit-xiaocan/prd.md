# PRD: 全面审查小蚕项目

## Goal / 用户价值

小蚕（活动监控+抢单+霸王餐抽奖系统）历经多次迭代（07-14 起频繁任务化开发，至今 30+ 个已归档任务），缺少一次系统性回看。用户要求做一次**全面审查**，覆盖前后端与生产线上行为，目标不是改代码，而是摸清：哪些地方有潜在 bug、哪些功能标了 done 但实际有缺口/半成品、哪些线上任务在静默失败或长期异常被吞掉。产出一份**问题清单（带严重度与位置）**，作为后续逐项修复的输入。

本次审查**只诊断、不修**。修复由后续单开任务承接。

## 范围（三维度，各对应一个 child）

### Child 1 — 代码质量/潜在 Bug（`07-20-audit-code-quality`）
后端 Java + 前端 Vue3 源码的正确性扫描：
- 并发/线程安全：cron 调度（MonitorCronScheduler/GrabCronScheduler）、多账号轮询换号、共享状态。
- 事务/一致性：service 写操作 `@Transactional` 覆盖、`updateById` 忽略 null 坑、去重防重键选择。
- 边界与异常吞掉：NPE、null 兜底、空集合、try/catch 吞异常、HTTP 调用失败处理。
- 去重防重逻辑：STORE_ACTIVITY 当天去重 / STORE_KEYWORD N 分钟窗 / MINIMUM_PAY N 分钟窗 / SINGLE 模式活动级成功防重 / 抢单防重，跨模块一致性。
- 资源/性能：N+1 查询、循环内查库、连接池/内存（生产 -Xmx256m 约束）。

### Child 2 — 功能完整性/缺口（`07-20-audit-feature-completeness`）
对照 README 更新记录与各模块声明的功能，核每个功能是否真实现完整：
- README「todo」未勾选项（"以及再次通知"）的真实状态。
- 各监控类型（STORE_ACTIVITY/STORE_KEYWORD/MINIMUM_PAY）配置→调度→抓取→过滤→推送→历史 闭环是否完整。
- 抢单：手动/cron/一次性精确执行、多账号多平台优先级轮询、换号降级、抢到/失败推送、饭票校验。
- 霸王餐抽奖：批量多账号、App 登录态契约、运行日志。
- 登录态统一池：抢单+霸王餐合并、前端选择框、PUT location。
- 代理 IP 池、推送按地址路由、商家黑名单、3km 筛选。
- 半成品/TODO/FIXME 注释、被注释掉的功能、前端有入口但后端无实现的接口。

### Child 3 — 生产运行健康（`07-20-audit-prod-health`）
SSH+日志核线上真实行为（`ssh root@121.91.175.192`，读 `/opt/xiaocan/logs/info.log`、systemd）：
- 各 cron 监控任务是否真按配置执行、有无长期静默失败。
- 抢单/抽奖任务实际触发与命中情况、有无长期无命中（可能逻辑坏）。
- 异常/ERROR 日志分布与频次、有无反复吞掉的异常栈。
- systemd 服务稳定性、重启次数、内存（-Xmx256m）是否接近上限、HikariCP 连接。
- 前端线上页面可访问性（nginx + dist 部署）、`/api/` 反代是否通。
- 与 README/spec 声明行为的偏差（线上实际 vs 声明）。

## 验收标准（parent，跨 child）

- [ ] 三个 child 各自产出**问题清单**（带 `file:line` 或日志证据、严重度 P0/P1/P2/P3、简述、复现/触发条件）。
- [ ] 三份清单汇总到 parent 的 `audit-findings.md`，按严重度排序、去重（同一问题多维度命中只保留一条并标注来源维度）。
- [ ] 每条问题标注**是否需后续修复**与**建议归属模块**，作为后续开修复任务的依据。
- [ ] 明确列出「审查中确认正常/无问题」的模块，避免只记问题造成「处处有 bug」的错觉。
- [ ] 全程**不改代码、不改生产配置**；只读源码、只读日志、只读 DB（必要时 SELECT）。

## Out of Scope

- 修复任何发现的问题（另开任务）。
- 性能压测、安全渗透、风控对抗深挖（本次仅列已知风险，不做对抗设计）。
- 逆向 X-Ashe 签名机制的再验证（已落地，作为既定事实）。
