# Design — 小蚕平台全量测试方案

## 测试方法论
两层观测、驱动同一真实登录态浏览器：
- **前端驱动**：Browser Relay（`127.0.0.1:18795`）控制已登录 Chrome。
  - `tabs` 定位标签 → `navigate` 进入模块 → `snapshot` 读页面结构 → `click`/`type`/`scroll` 操作 → `snapshot`/`screenshot` 复核。
  - `console` 读页面报错，`network` 读请求/响应（敏感头脱敏），定位前后端边界问题。
- **后端观测**：ssh `root@121.91.175.192`，`journalctl -u xiaocan -n N --no-pager` / `-f` 看后端日志
  （服务清单见 server-services.md，nginx 反代 xiaocan.20030704.xyz → 127.0.0.1:10234）。
  前端请求与后端日志对齐时间戳，判断接口行为。
- **数据不稳定应对**：页面「需多次刷新才出数据」→ 用 `navigate` 重复加载或 `eval` 触发重新请求，
  记录出现/不出现次数与耗时，作为性能/稳定性优化点。

## 测试顺序（由低风险到高风险）
1. 设置 `/settings` — 看代理/登录态/账号配置现状（只读优先，为后续模块定基线）。
2. 地址管理 `/location` — 地址是其他模块依赖项；增删改用可恢复测试数据。
3. 首页活动列表 `/` — 依赖地址；测搜索/排序/3km过滤/刷新稳定性。
4. 卡券查询 `/grab-card` — 饭票数量查询，只读为主。
5. 监控管理 `/monitor` — 配置项增删改，可恢复。
6. 通知记录 `/notify-history` — 只读历史。
7. 抢单记录 `/grab-history` — 只读历史。
8. 抢单 `/grab` — 风险最高，最后测；先验证配置/饭票校验拦截，实际抢单触发前评估并报告风险。

## 抢单模块风险评估
- 用户已确认：配置了代理 API，**坚持实际下单**。
- 策略：先测「配置保存」「饭票校验拦截」「cron/一次性定时校验」；
  再选一个可抢活动实际触发抢单，全程 ssh 同步看后端日志、前端 network 看小蚕接口响应与风控状态。
- 实际下单可能触发小蚕风控/WAF/封 IP 数小时——这是测试要观察并记录的现象本身。
- 报告中如实标注触发深度、风控响应、代理是否生效。

## 问题分级
- P0 阻断：功能完全不可用/报错。
- P1 严重：核心路径可用但有数据错乱/风控误触/严重不稳定。
- P2 一般：边界/体验/性能问题（含需多次刷新才出数据）。
- P3 优化建议：可用但可改进。

## 产物
- `report.md`：分模块测试结果 + 问题清单 + 优化建议。
- 测试中发现的可立即修复的小问题，在报告标注后由用户决定是否进 execute 阶段修复。
